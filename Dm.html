Field Types

Data model fields are classified according to their implementation, technology platform, and application usage as shown in Figure 12 on page 306.

  • Topology
  • Private management
  • Presentation

Each class or object contains fields which are either defined on a particular class or inherited from its parent class. See “Field Inheritance” on page 6 for a description of field inheritance.

Topology Fields

Topology fields are common OSI fields inherited from OSI classes. These fields are shared by Tivoli applications on RODM and consist of two different kinds of fields:

  • Implementation-independent fields
  • RODM application-independent fields

See Figure 12 on page 306 as to where the implementation-independent fields ▌1▐ and the RODM application-independent fields ▌2▐ fit in the classification of fields

Field Names for Topology Fields

The descriptive name of each field is a single word or a concatenated word string with the first word of the string never being capitalized, but each subsequent word being capitalized. The topology field descriptive names represented by a single word are italicized to avoid misunderstandings.

Each implementation-independent field has an ASN.1 object identifier in addition to its descriptive name. The ASN.1 object identifier is used as its field name when defining the field to RODM. For example:

1.2.124.360501.1.209 -- attachedCircuitList --

RODM application-independent fields, such as the memberOf field, do not have ASN.1 object identifiers. Their descriptive names are used as the field name when defining the fields to RODM

Implementation-independent Fields

Most topology fields are implementation-independent fields that are derived fromexisting and emerging open standards. These implementation-independent fields, shown as ▌1▐ in Figure 12 on page 306, are necessary for topology management in any technology platform.

RODM application-independent fields

The remaining topology fields are specific in their implementation for a particular data storage technology. In the case of the SNA topology manager and MultiSystem Manager, these RODM application-independent fields, shown as ▌2▐ in Figure 12, complement the implementation-independent fields to enable and preserve the integrity of the model within the constraints of the technology.

Private Management Fields

The private management fields are application-specific fields, defined by each manager, that contain operational data to enhance the management and monitoring of the designated class of resources needed by that manager.


These fields are classified as RODM-dependent fields that are specific to each manager application. See reference ▌3▐ in Figure 12. These fields are specifically defined for each topology manager to implement its topology management function. Their use is specific and private in the sense that another application might not define the same set of fields in its implementation of the management function. The first character of a private management field name is always uppercase

Presentation Fields

The presentation fields are classified as fields within RODM that enable each topology manager to graphically display its views using GMFHS. See reference ▌4▐ in Figure 12. MultiSystem Manager, SNA Topology Manager, and GMFHS all use these common presentation fields. The use of these fields is specific and private to GMFHS in the sense that another feature similar to GMFHS might not implement the same display function and might define a different set of presentation fields altogether.

Field Restrictions

The following restrictions apply to fields that are used by the SNA topology manager

  • Do not install query methods on any fields that are supplied by the NetView product. The SNA topology manager uses the RODM EKG_QueryMultipleSubfields function which does not trigger query methods.
